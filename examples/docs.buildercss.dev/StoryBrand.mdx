# StoryBrand Framework: docs.buildercss.dev

## 1. Character (The Hero)

**Who is our customer?**

The Frontend Developer who wants to build beautiful websites faster using BuilderCSS's AI-powered approach.

**What do they want?**
- Clear, comprehensive documentation
- Quick start guides that actually work
- Component examples they can copy
- Best practices for BuilderCSS patterns

## 2. Problem

### External Problem (Tangible)
"I want to use BuilderCSS but I need to understand how it works first."

### Internal Problem (Emotional)
"I'm not sure if I'm using this framework correctly. Am I missing something?"

### Philosophical Problem (Why It's Wrong)
Great tools shouldn't require hours of documentation spelunking to use correctly.

### The Villain
**Incomplete Documentation** - The bane of every developer trying to learn a new framework.

## 3. Guide (The Brand)

**Position:** We've created documentation that respects your time and gets you building fast.

**Empathy:** We've been lost in bad docs. Every example here actually works. Every explanation is tested.

**Authority:**
- Complete API reference
- Real-world code examples
- Interactive component playground
- Search that actually finds things

## 4. Plan (The Path)

### Step 1: Quickstart
Follow the 5-minute quickstart and have your first BuilderCSS page running.

### Step 2: Learn Concepts
Understand the core concepts: layouts, containers, sections, views, and pages.

### Step 3: Build
Use the component library and patterns to build real projects.

## 5. Call to Action

**Direct CTA:** Read the Quickstart

**Transitional CTA:** Browse Components

## 6. Success (What's at Stake)

**Success looks like:**
- Understanding BuilderCSS deeply
- Building beautiful sites confidently
- Following best practices automatically
- Shipping faster with fewer bugs

**Transformation:**
From: Developer unsure about the framework
To: Developer building confidently with BuilderCSS

## 7. Failure (What We Help Them Avoid)

**Failure looks like:**
- Hours lost to documentation gaps
- Anti-patterns from poor examples
- Frustration giving up on the framework
- Shipping code that fights the framework
